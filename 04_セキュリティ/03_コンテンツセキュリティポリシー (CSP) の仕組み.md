### CSPの仕組み

サーバーがブラウザに対して、HTTPレスポンスヘッダーの Content-Security-Security-Policy を使って指示を出します。

#### 具体的な設定例

例えば、以下のようなヘッダーを送信したとする。

`Content-Security-Policy: default-src 'self';script-src 'self' https://trusted.com;`

この命令をブラウザが受け取ると、以下のように動きます：

- `default-src 'self':` 画像やスタイルシートなどは、自分のサイト（自分と同じドメイン）からのみ読み込む。

- `script-src 'self' https://trusted.com: JavaScript`は、自分のサイトと trusted.com 以外からは絶対に実行しない。

もし攻撃者がインライン（HTMLの中に直接）で <script>alert('XSS')</script> と書き込んでも、CSPで許可されていなければブラウザはこれを無視する。

### CSPで制御できる主なリソース

CSPはJavaScriptだけでなく、ウェブページのあらゆる要素を細かく制限できる。

| ディレクティブ  | 制御する対象                                                                   |
| --------------- | ------------------------------------------------------------------------------ |
| script-src      | JavaScriptの読み込み・実行元を制限。                                           |
| style-src       | CSS（スタイルシート）の読み込み元を制限。                                      |
| img-src         | 画像の読み込み元を制限。                                                       |
| frame-ancestors | 自分のサイトを`<iframe>`で表示できるサイトを制限（クリックジャッキング対策）。 |
| connect-src     | FetchやXMLHttpRequest（通信）の宛先を制限。                                    |

### 導入のステップ：Report-Only モード

いきなり厳しい制限をかけると、意図せず自分のサイトの機能が動かなくなる（画像が表示されない、外部APIが叩けない等）ことがあります。

そこで便利なのが Content-Security-Policy-Report-Only ヘッダーです。

- 機能: ポリシーに違反してもブロックはせず、「もし本番運用だったらここで違反が起きていたよ」という通知だけを指定のURL（report-uri）に送ります。

- メリット: サイトを壊すことなく、安全に設定をテストできます。

### XSS対策としての決定打「Nonce」と「Hash」

最近の推奨設定では、ドメイン指定（trusted.com など）よりも、さらに厳格な方法が使われます。

- Nonce（ナンス）: ページを読み込むたびに生成される「使い捨ての合言葉」です。`<script nonce="随机値">` のように合言葉が一致するスクリプトだけを実行します。

- Hash（ハッシュ）: スクリプトの内容そのものをハッシュ化し、「この内容のスクリプトだけ許可する」と指定します。
