## Sass (Syntactically Awesome Style Sheets)

Sass は CSS の「拡張版」である。ブラウザは Sass を直接理解できないため、ビルド時に標準の CSS へ変換（コンパイル）して利用する。

#### 仕組み

- 変数とネスト: `$primary-color: #333;` のように変数が使えたり、親子関係を入れ子（ネスト）で書けたりする。

- 再利用性: `@mixin` を使って、よく使うスタイルを関数のように使い回せる。

- 注意点: Sass 自体は「グローバル汚染」を解決しない。クラス名が被ればスタイルは上書きされる。そのため、BEM（`block__element--modifier`）のような命名規則とセットで運用されるのが一般的である。

## CSS Modules

CSS Modules は、「CSS ファイルを JavaScript のオブジェクトとして読み込む」仕組みである。主に Webpack などのビルドツールによって実現される。

#### 仕組み

- 普通の CSS ファイル（例: Button.module.css）を書く。

- JS 側で import styles from './Button.module.css' と読み込む。

- ビルド時に、クラス名が `Button_btn\_\_a1b2c` のような「一意な名前」に自動変換される。

#### メリット

- カプセル化: 他のコンポーネントと同じ `.btn` というクラス名を使っても、ビルド後の名前が重ならないため、スタイルが衝突しない。

- 学習コストが低い: 書き方は普通の CSS そのものである。

## CSS-in-JS (Styled Components / Emotion など)

CSS-in-JS は、その名の通り JavaScript の中に直接 CSS を書く 手法である。React エコシステムで非常に人気がある。

#### 仕組み

- テンプレートリテラル: JS のバッククォート（`）の中にスタイルを記述し、それを「スタイル付きコンポーネント」として定義する。

- 動的なスタイル変更: props を CSS に渡せるため、「ボタンが押されたら色を変える」といったロジックを、JS の変数を使って直接制御できる。

#### メリット

- 未使用コードの削除: コンポーネントが削除されれば、そのスタイルも自動的に消える。

- JS との親和性: スタイルとロジックを 1 つのファイル（`.js` / `.tsx`）にまとめられる。

## 比較

| 特徴           | Sass                   | CSS Modules            | CSS-in-JS                            |
| -------------- | ---------------------- | ---------------------- | ------------------------------------ |
| 実体           | CSS の高機能版         | 標準 CSS + ビルド設定  | JavaScript 経由の生成                |
| 名前の衝突     | 命名規則で防ぐ必要あり | 自動で回避される       | 自動で回避される                     |
| 動的な変更     | クラスの付け替えが必要 | クラスの付け替えが必要 | JS の変数で直接制御                  |
| パフォーマンス | 高い（静的ファイル）   | 高い（静的ファイル）   | 実行時の計算コストが発生する場合あり |

## 使い分け

- 大規模でロジックが複雑な React プロジェクト: JS と密接に連携できる`CSS-in-JS` が強力である。

- パフォーマンス重視、またはデザイナーと分業する場合: 慣れ親しんだ書き方で、かつ名前が衝突しない `CSS Modules` がバランス良い。

- 静的なサイトや、特定のフレームワークに依存したくない場合: シンプルな `Sass` が良さそう。
